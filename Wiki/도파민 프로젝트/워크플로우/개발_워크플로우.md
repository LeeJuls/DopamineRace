# 개발 워크플로우

---

## 기본 사이클

```
1. 계획 (leader → SPEC 작성 → 오너 컨펌)
   ↓
2. 구현 (client → Unity MCP 검증)
   ↓
3. 검증 (qa → TC 실행 → PASS/FAIL)
   ↓
4. 히스토리 작성 (leader → HIST 문서)
   ↓
5. 오너 컨펌 → 커밋 → (오너 지시 시) Push
```

---

## Git 규칙

| 항목 | 규칙 |
|------|------|
| 커밋 접두사 | `[C]` Claude, `[L]` 오너, `[UI]` 디자이너 (병기 가능: `[C][DOC]`) |
| Push | **오너 명시 요청 시에만** |
| 브랜치 | main + 세션별 `claude/xxx` 워크트리 |

---

## 워크트리 규칙 (중요)

> ⚠️ `cp worktree → main` 직접 복사 **금지** — regression 유발 (2026-03-06 사고)

**동기화 방법**: `git cherry-pick` 또는 `git merge`

**동기화 전 확인**:
```bash
git diff HEAD..claude/<브랜치> -- <파일>
# main이 최신인지 확인 후 진행
```

**SceneBootstrapper partial 6개**는 항상 함께 비교/동기화.

---

## 세션 시작 체크

```bash
git fetch --all && git log --oneline -5 main
# 새 커밋 있으면 git merge main
```

---

## 푸시 절차 (3단계)

```bash
# 1. 워크트리 브랜치 → origin 푸시
git push origin claude/<브랜치>

# 2. main checkout → 워크트리 머지 → origin/main 푸시
git checkout main
git merge claude/<브랜치>
git push origin main

# 3. 워크트리로 돌아와 main 동기화
git checkout claude/<브랜치>
git merge main --no-edit
```

## 디버깅 원칙

- 근본 원인 없이 수정 금지
- 에러 재현 → 로그/스택 → 최근 변경 추적 → 가설 수립
- 한 번에 하나만 수정, 결과 확인 후 다음
- **3회 실패 시 중단** → 아키텍처 재검토 + 오너 호출
