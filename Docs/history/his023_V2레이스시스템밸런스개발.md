## 5. GameSettings.asset 직렬화 문제 수정

**문제**: Unity ScriptableObject는 코드 기본값 변경 시 기존 `.asset`의 직렬화 값을 유지함
→ `v2_exhaustFloor`가 에셋에 `0.75`로 고정 (코드 변경 0.70 무시됨)

**수정**: `Assets/Resources/GameSettings.asset` 직접 수정
```yaml
v2_exhaustFloor: 0.75  →  v2_exhaustFloor: 0.7
```

**확인된 에셋 상태 (수정 후)**:
- `v2_drainBaseRate` — 에셋에 없음 → 코드 기본값 1.0 정상 적용
- `v2_phaseCoeff_*` — 에셋에 없음 → 코드 기본값 정상 적용
- `v2_exhaustFloor` — 에셋 직렬화 값 0.70 ✅

## 6. EN 순위 서수 복원

**파일**: `Assets/Scripts/Utility/Loc.cs`

**문제**: `str.hud.rank` EN 포맷이 `{0}.`이라 "8." 형태만 출력
**수정**: `Loc.GetRank()`에 EN 서수 생성 로직 추가

```csharp
if (CurrentLang == "en")
{
    int abs = Mathf.Abs(rank);
    string suffix;
    if (abs % 100 >= 11 && abs % 100 <= 13)
        suffix = "th";
    else
        suffix = (abs % 10) switch { 1 => "st", 2 => "nd", 3 => "rd", _ => "th" };
    return rank + suffix + ".";
}
```

→ `1st.` `2nd.` `3rd.` `8th.` 형태로 복원

## 7. 실기기 테스트 — V2 구조적 한계 발견

### 현상
- **2바퀴**: 1바퀴 후 HP ~60%, 2바퀴 후 HP 20~32% (HP 잔존)
- **3바퀴**: 2바퀴 후 HP ~25-40%, **3바퀴 완료 시 전원 HP 0%**
- **5바퀴**: 3바퀴에서 전원 HP 0% → 4-5바퀴는 exhaustion 상태로 순위 고착화

### 수치 분석
```
실제 HP 소모/초 = drainBaseRate × speedRatio² × globalSpeedMultiplier
               = 1.0 × ~1.0 × 4 = 4 HP/실제초

5바퀴 실제 완주시간 ≈ 120초
→ 총 HP 소모 추산: 480 HP (maxHP=110 대비 4배 초과)
→ 3바퀴 (~30초 실제) 시점에 이미 HP 고갈
```

**근본 원인**: `globalSpeedMultiplier=4`를 드레인에 그대로 곱함
→ V2 설계 시 이 수치를 고려하지 않음
→ 백테스트는 순위 결과만 검증 (HP 잔존량 미검증)

### 추입 우세 문제
- Reckoner: 스프린트 75% 시점 → HP 최대 보존 → 스프린트 속도 최고
- **단거리(2바퀴)에서도 2-3위** → 설계 의도(단거리=Runner 우세)와 불일치
- 원인: HP 보존이 속도에 직결되는 구조 + 모든 거리에서 Reckoner가 유리

## 8. V3 방향 논의 (결론)

### 핵심 결정
- V3는 **V2와 완전히 분리된 `v3_*` 설정 블록** 사용
- GameSettings 전체 수치(`globalSpeedMultiplier`, `maxHP`, 랩 수 등)를 **처음부터 함께 고려**

### 제안된 V3 핵심 아이디어: 거리 노멀라이제이션
```
drainRate = (maxHP × targetExhaustRatio)
          / (lapCount × avgLapTime × globalSpeedMultiplier)
```
→ 어떤 거리든 레이스의 **80% 시점에 HP 소진** 보장
→ 마지막 20% 구간은 항상 역전 가능한 상태로 유지
